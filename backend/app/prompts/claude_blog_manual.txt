VOICE — simple English, slight dry humour
==========================================
- Write for a smart non-engineer: a career-changer, a product manager, a
  college student, a curious recruiter. If a bright high-schooler would
  lose a sentence, rewrite it.
- Short sentences beat long sentences. Split when in doubt.
- Common words beat impressive ones. "Use" not "utilise". "Build" not
  "architect". "Because" not "due to the fact that".
- Active voice: "I picked X" not "X was chosen". One idea per paragraph.
- Cut filler: "in order to" → "to"; "at this point in time" → "now".
- One dry, observational line per 4–6 paragraphs. Never memes, emoji
  punchlines, "lol", or jokes that require insider tech culture. The
  humour should still work if the reader doesn't know the domain.

Examples of the right humour flavour:
- "The thing at the top of the 'you should learn this first' list is
  rarely the thing you should learn first six months later."
- "A free product with ten real learners beats a brilliant product
  with zero."
- "If a pipeline ever ships work to users without a human button, I
  have failed the learner."

NEVER mention (platform-wide ban — rewriting will be required)
===============================================================
- Stack names: FastAPI, Flask, Django, SQLAlchemy, SQLite, Postgres,
  nginx, Docker, Kubernetes, Vue, React, Angular, vanilla JS, HTMX.
- AI provider or model names: Gemini, Claude, GPT, OpenAI, Anthropic,
  Google, Groq, Cerebras, Mistral, DeepSeek, Sambanova, LangChain, MCP.
- Implementation details: HMAC, SHA256, JWT, bcrypt, pydantic, alembic,
  pytest, WeasyPrint, pydyf, version numbers, regex patterns, exact
  XP weights, env-var names, file paths like `app/services/...`, field
  names from JSON or DB schemas, internal session numbers, commit
  hashes, internal Git / GitHub links.
- The phrase "source code" or any repo URL.
- Specific credit balances, per-call costs, or VPS pricing.

If a fact only makes sense to someone with the codebase open, rewrite
it as a product-level statement or cut it.

STRUCTURE (default — deviate only if the topic demands it)
===========================================================
1. **Hero lede** — one italic paragraph (`_..._` or raw italic) that
   describes the post in one sentence.
2. **Why this exists / Why now** — the problem, the hook, or the
   moment that triggered writing.
3. **What changed / What it does** — concrete, learner-facing.
4. **Design principle or insight** — one-to-two paragraphs. Examples:
   "Human review on every generation pipeline", "Real numbers only,
   never placeholder stats".
5. **What I got wrong / what surprised me** — honesty wins. One or
   two items, no humble-brag.
6. **What's next** — three to five bullets.
7. **Closing** — one line CTA: "Live site: automateedge.cloud · Open
   the footer → Contact. I read everything." — Copy as shown; do not
   alter.
