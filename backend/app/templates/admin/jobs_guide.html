2. Your Daily 3-Step Workflow

Everything else runs automatically. These are the only 3 things you do.

1.Generate summaries — run /summarize-jobs --status draft --limit 50 in Claude Code (~2 min).
2.Open /admin/jobs — handle each tab in order (auto-skipped → Tier-2 → Tier-1 bulk).
3.Publish or reject each job. Always pick a reject reason.

Tab-by-tab: what to do

Filter (admin_notes)What it meansYour action

tier2-liteLightweight extraction (PhonePe, Groww, CRED, Notion, etc.)Review individually → publish or reject
(empty)Full Tier-1 enrichment (Anthropic, Scale, xAI, Cohere, etc.)Bulk Publish after spot-check (2–3 jobs)
enrichment failed: ...AI provider errorRetry or reject low_quality

3. Publishing — what to check

Every job must have an Opus summary card before publishing. Flash no longer generates summaries. If a job is missing one, run /summarize-jobs --id <ID> first.

Quick checklist — approve only if ALL true

  • Company is real AI/ML employer (not staffing firm / course-seller)
  • Role is genuinely AI/ML
  • posted_on is within last 45 days
  • designation enum matches the actual role
  • tldr reads naturally (NOT copied from JD — SEO penalty)
  • apply_url resolves (click "Test link")
  • Summary card is present (run /summarize-jobs --id N if missing)
  • No PII / email / phone leaked in description_html

After you click Publish

AUTO The system stamps review metadata, updates source stats, and pings IndexNow. Nothing else for you to do.

4. Rejecting — pick the right reason

ReasonUse when
fakeScam, ghost job, unverifiable company
expiredJD says "closed" or posted_on > 45d
off_topicNot AI/ML (devops, generic backend, sales, HR)
duplicateAlready published via another source (same company + title)
low_qualityJD too vague, no skills, enum violations
Why the reason matters: AUTO The system feeds the last 45 days of rejection reasons back into the AI extraction prompt. Your reason makes the extractor smarter over time. Rejecting without a reason breaks this loop.

7. AI Classification — Defense Layers

The system uses 10 layers (RCA-026 + Waves 1–5) to defend against false positives like the PhonePe "Manager, Legal" incident, where Gemini confused LLM (Master of Laws degree) with LLM (Large Language Model). All filters bias toward rejection — better to surface a borderline job as no-topic for your review than to mislabel a non-AI role as AI.

Filter pipeline order

1.Title pre-filter — ~120 patterns across 21 non-AI categories (sales engineer, recruiter, policy analyst, UX designer, etc.)
2.JD body cluster scanner — flags JDs with ≥2 hits from 9 non-AI clusters (legal/HR/finance/sales/marketing/design/recruiting/IT/creative/policy) when AI-intensity score < 5
3.Bare-verb title gate — "Manager, X" / "Director, Y" / "Head of Z" without AI anchor word AND low intensity = auto-skip
4.AI-intensity scoring — weighted 3-tier (STRONG x3 + MEDIUM x2 + WEAK x1, deduped, boilerplate stripped, requirement-phrases neutralized). Threshold: 5
5.Self-rejection prompts — Gemini instructed to return topic:[] for sales/marketing/legal/HR/etc. roles even at AI-first companies
6.Evidence-span validation — Gemini must cite a JD substring for each topic; we verify the quote exists and check it against forbidden patterns (LLB/Master of Laws for "LLM", workplace safety for "Safety", user research for "Research")
7.Topic-anchor check — each assigned topic must have an anchor phrase elsewhere in the JD body
8.Designation↔topic consistency — designation == "Other" forces topic = []; AI-adjacent designations (AI PM, Solutions Architect, Developer Advocate, Prompt Engineer) capped at 1 topic
9.Per-source rejection-rate alarm — auto-disables sources with >40% admin reject rate over last 30d (min 20 reviewed rows)
10.Weekly Opus audit — 1% of Tier-1 published jobs flagged for manual Claude Code review (no API spend — uses Claude Max in VS Code)

What you'll see in the queue

If you see false positives slipping through: log it. The next backfill (scripts/backfill_rca026_non_ai.py --apply) catches historical rows; the rejection-rate alarm and intensity histogram surface drifting sources before they become a problem.

8. Opus Audit — Weekly Workflow

Every Monday 04:30 UTC the scheduler picks ~1% of Tier-1 published jobs (clamped 1–20) and stamps them as audit-pending. You see an amber banner at the top of /admin/jobs: "N pending Opus audit [COPY PROMPT]".

Step-by-step

1.Click COPY PROMPT in the banner. The Claude Code prompt (with all pending jobs JSON-embedded) is copied to clipboard.
2.Open Claude Code in VS Code (with your Claude Max subscription — no API spend).
3.Paste the prompt. Claude returns a JSON array with one entry per job: {job_id, agreed, opus_topic, opus_designation, notes}.
4.POST that JSON back as {"results":[...]} to /admin/jobs/api/audit-submit (curl from your shell or the VS Code REST extension).
5.Disagreements stamp OPUS-AUDIT mismatch (date): notes on the job's admin_notes for follow-up review.

Submission example

curl -X POST https://your-domain/admin/jobs/api/audit-submit \
  -H "Cookie: session=..." -H "Content-Type: application/json" \
  -d '{"results":[
    {"job_id":20,"agreed":true,"opus_topic":["RL","Safety","Research"],
     "opus_designation":"Research Scientist","notes":""},
    {"job_id":95,"agreed":false,"opus_topic":[],
     "opus_designation":"Other","notes":"Customer support, not AI"}
  ]}'
Why no API spend? Claude Max in VS Code is a fixed monthly subscription, not metered API. The audit prompt + ~10 jobs fits in one Opus context window with room to spare.

If you skip a week

Pending count accumulates. Cooldown is 90 days — once you submit a verdict, that job won't be re-audited for 90 days. So skipping a week just delays detection of any drift on those particular jobs by one week.

9. Never do this

  1. Publish without a summary card — run /summarize-jobs first
  2. Publish without reading the JD — even Tier-1 can have extraction errors
  3. Bulk-reject without picking reasons — breaks the extractor feedback loop
  4. Edit posted_on — it's the source's truth, not ours
  5. Manually re-enable a probe-disabled board without verifying the URL is back
  6. Approve a Tier-2 job without checking the company's own website
  7. Approve > 20 jobs in one click without spot-checking
  8. Manually assign a topic to a role marked topic = [] by the validator without reading the JD — the validator strips topics for a reason
  9. Submit Opus audit results without reading them — mismatches stamp admin_notes; pasting blindly poisons your own queue

Opus audit banner staleness

When the amber banner says "last audit YYYY-MM-DD", that date should be within the past 7 days. If it shows "no prior audit on record", you have never submitted audit results — run the workflow in Section 8 today.
