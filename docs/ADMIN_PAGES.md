# Admin Pages Reference

> Complete documentation of all admin-facing pages. Read this before modifying or extending admin UI.

## Navigation

All admin pages share a unified sticky header with the site logo, brand name, and these links:

| Link | Route | Router File |
|------|-------|-------------|
| Dashboard | `/admin/` | `routers/admin.py` |
| Users | `/admin/users` | `routers/admin.py` |
| Templates | `/admin/templates` | `routers/admin.py` |
| Pipeline | `/admin/pipeline/` | `routers/pipeline.py` |
| Topics | `/admin/pipeline/topics` | `routers/pipeline.py` |
| Settings | `/admin/pipeline/settings` | `routers/pipeline.py` |

**Hidden page** (accessible but not in nav):
- Proposals: `/admin/proposals` — curriculum proposal review (legacy, rarely used)

**Auth:** All admin pages require `get_current_admin` dependency (checks JWT + `is_admin=True`).

## Page 1: Dashboard (`/admin/`)

**File:** `backend/app/routers/admin.py` → `admin_dashboard_page()`

**Purpose:** Overview of platform health and user activity.

**Stats displayed:**
- Total Users — count of all users in DB
- DAU / WAU / MAU — distinct users with sessions issued in last 1/7/30 days
- Dead Links — templates with `consecutive_failures > 2` in `link_health` table

**Recent Signups table:** Last 10 users created in past 7 days (ID, email, name, created date).

**Data sources:** `User`, `Session`, `LinkHealth` models. All queries are real-time (no caching).

**API endpoint:** `GET /admin/api/dashboard` — returns JSON with same stats.

---

## Page 2: Users (`/admin/users`)

**File:** `backend/app/routers/admin.py` → `admin_users_page()`

**Purpose:** Searchable, paginated user list.

**Features:**
- Search by email or name (SQL `CONTAINS` query)
- Pagination: 20 per page with Prev/Next links
- Columns: ID, Email, Name, Provider (google/otp), Admin flag, Created date

**API endpoint:** `GET /admin/api/users?q=&page=1&per_page=20`

**No mutations** — this page is read-only. No delete/edit/promote actions exposed.

---

## Page 3: Templates (`/admin/templates`)

**File:** `backend/app/routers/admin.py` → `admin_templates_page()`

**Purpose:** View all curriculum templates and generate new ones via AI.

**Features:**
- **Generate New Template** form: Topic (text), Duration (3/6/9/12 months), Level (beginner/intermediate/advanced)
  - Calls `POST /admin/api/generate-template` → `curriculum_generator.generate_curriculum()` + `save_curriculum_draft()`
  - Takes 15-30 seconds (AI generation)
  - Saves JSON to `curriculum/templates/{key}.json`
  - Clears template LRU cache so new template appears immediately
- **Templates table:** Title, Goal, Level, Duration, Weeks, Checks, Delete button
  - Default generalist templates (3 built-in) cannot be deleted
  - AI-generated templates can be deleted

**API endpoints:**
- `GET /admin/api/templates` — list all templates with metadata
- `POST /admin/api/generate-template` — generate new template via AI
- `DELETE /admin/api/templates/{key}` — delete a template (not generalist_*)

**Template storage:** JSON files in `backend/app/curriculum/templates/`. Loaded via `loader.py` with `@lru_cache(maxsize=8)`.

## Page 4: Proposals (`/admin/proposals`)

**File:** `backend/app/routers/admin.py` → `admin_proposals_page()`

**Purpose:** Review curriculum update proposals generated by quarterly sync.

**Features:**
- List all proposals (ID, Source Run, Status, Notes, Created, Actions)
- Pending proposals have Apply/Reject buttons
- Status lifecycle: `pending` → `applied` or `rejected`
- Reviewer ID and timestamp recorded on action

**API endpoints:**
- `GET /admin/api/proposals` — list all proposals
- `POST /admin/api/proposals/{id}/apply` — mark as applied
- `POST /admin/api/proposals/{id}/reject` — mark as rejected

**Note:** This page is legacy from the manual quarterly sync flow. The new auto-curriculum pipeline (Pages 5-7) largely replaces this workflow.

## Page 5: Pipeline Dashboard (`/admin/pipeline/`)

**File:** `backend/app/routers/pipeline.py` → `pipeline_dashboard_page()`

**Purpose:** Overview and manual controls for the auto-curriculum pipeline.

**Stats displayed:**
- Total Topics — all discovered topics in DB
- Pending Review — topics with `status=pending`
- Approved — topics with `status=approved`
- Generated — topics with `status=generated`
- Templates — total template files on disk
- Budget Used — `tokens_used_this_month / max_tokens_per_run` as percentage

**Action cards (3):**

| Card | Button | What it does | Service |
|------|--------|-------------|---------|
| Topic Discovery | Run Discovery Now | AI researches trending AI/ML topics, saves to DB as pending | `topic_discovery.run_discovery()` |
| Batch Generation | Generate All Pending | Generates 5 template variants per approved topic | `batch_generator.run_batch_generation()` |
| Content Refresh | Run Refresh Now | Checks all resource URLs + AI reviews template currency | `content_refresh.run_content_refresh()` |

Each button shows processing status and auto-refreshes the page on success.

**API endpoints:**
- `POST /admin/pipeline/api/run-discovery` — trigger topic discovery
- `POST /admin/pipeline/api/run-generation` — trigger batch generation
- `POST /admin/pipeline/api/run-refresh` — trigger content refresh

**Background scheduler:** `pipeline_scheduler.py` runs these automatically based on configured frequencies (check interval: 6 hours).

## Page 6: Topics (`/admin/pipeline/topics`)

**File:** `backend/app/routers/pipeline.py` → `pipeline_topics_page()`

**Purpose:** Review, approve, reject, and manage AI-discovered topics.

**Features:**
- **Status filter tabs:** All, Pending, Approved, Generating, Generated, Rejected
- **Pagination:** 50 per page with Prev/Next
- **Columns:** ID, Topic (with category/subcategory), Justification (truncated), Confidence Score, Status badge, Templates generated count, Discovered date, Actions

**Topic lifecycle (state machine):**
```
pending → approved → generating → generated
   ↓         ↓
rejected   rejected
   ↓
approved (re-approve)
```

**Actions per status:**
- Pending: Approve, Reject, Delete
- Approved: Reject, Delete
- Generating: (no actions — in progress)
- Generated: Delete
- Rejected: Re-approve, Delete

**API endpoints:**
- `GET /admin/pipeline/api/topics?status=&page=1&per_page=50` — list topics
- `POST /admin/pipeline/api/topics/{id}/approve` — approve topic
- `POST /admin/pipeline/api/topics/{id}/reject` — reject topic (accepts `notes` in body)
- `DELETE /admin/pipeline/api/topics/{id}` — delete topic

**Deduplication:** Topics are deduplicated by `normalized_name` (unique column). Re-running discovery won't create duplicates.

## Page 7: Settings (`/admin/pipeline/settings`)

**File:** `backend/app/routers/pipeline.py` → `pipeline_settings_page()`

**Purpose:** Configure the auto-curriculum pipeline behavior.

**Settings (all stored in `curriculum_settings` singleton row):**

| Setting | Type | Default | Description |
|---------|------|---------|-------------|
| Max Topics Per Discovery | int (1-50) | 10 | How many topics AI should discover per run |
| Discovery Frequency | enum | monthly | How often auto-discovery runs (weekly/monthly/quarterly) |
| Auto-approve Topics | bool | false | If true, discovered topics skip pending → go straight to approved |
| Auto-generate Variants | bool | true | If true, auto-generates templates after discovery when auto-approve is on |
| Research Model | enum | gemini | AI model for deep research (gemini/groq) |
| Formatting Model | enum | groq | AI model for cheap classification/triage (groq/gemini) |
| Max Tokens Per Month | int (0-1M) | 50,000 | Monthly token budget for pipeline AI calls |
| Refresh Frequency | enum | quarterly | How often content refresh runs (monthly/quarterly) |

**Read-only display:**
- Current Usage: `tokens_used_this_month / max_tokens_per_run` with percentage

**Validation:** `PipelineSettingsUpdate` Pydantic model enforces types, ranges, and enum values.

**API endpoints:**
- `GET /admin/pipeline/api/settings` — get current settings
- `POST /admin/pipeline/api/settings` — update settings (partial update, only provided fields)

### CSRF Protection

All state-changing endpoints call `_check_origin(request)`:
- Parses Origin/Referer header
- Compares hostname with strict equality (not substring)
- Rejects requests with missing Origin+Referer headers

### Security Notes

- All endpoints require `get_current_admin` (JWT + is_admin check)
- HTML output uses `html.escape()` on all DB-sourced strings
- Settings input validated via Pydantic model
- AI prompts sanitize DB-sourced strings (truncate + strip newlines)
- Link health checker blocks SSRF (private IPs, metadata endpoints)
- See `docs/RCA.md` entries 011-014 for security fixes applied
