AI Enrichment Engine

01Why an enrichment layer exists

Normalization gives you clean, canonical records. Enrichment is what turns those records into decisions — adding context, classification, scoring, and recommendations that no single source could produce alone. It's the layer where external data and an LLM combine to answer the question "what does this record actually mean, and what should I do about it?"

Done well, enrichment is the most valuable layer in the platform. Done badly, it's the most expensive — runaway token bills, hallucinated outputs, latency spikes, and cache hit rates that nobody measures. The difference between the two outcomes is almost entirely a matter of architecture, not prompt engineering.

The four invariants that keep enrichment sane

1. Every LLM call is preceded by a cache check.
2. Every LLM call is preceded by a budget check.
3. Every LLM output is validated against a strict schema before storage.
4. Every LLM output includes its own reasoning trail for audit.

02Core principles

Cache before you call

Redis is checked first on every single request. If the value is hot, the LLM is never invoked. Cache TTLs are tuned per record type — volatile records expire fast, stable records live for days.

Budget before you spend

Every tenant has a monthly token budget per module. Every call checks the budget before issuing the request. At 80% of budget the system warns; at 90% it falls back to a cheaper model; at 100% it hard-stops.

Schema before you trust

The LLM is instructed to return JSON matching a strict schema, and the response is validated at runtime. If validation fails, the result is rejected — never coerced, never stored as free text.

External signals before LLM tokens

Free or cheap external APIs (lookups, public databases, internal services) are called in parallel before the LLM. The LLM reasons over the consolidated provider output rather than wasting tokens looking things up.

Reasoning is part of the output

Every enrichment includes evidence sources, score justification, and uncertainty factors. A score with no reasoning trail is worse than no score at all — it's a number nobody can defend.

Async by default, sync by exception

Bulk enrichment runs through a batch queue. Real-time enrichment is reserved for human-triggered actions (an analyst clicks a record). The two paths share the same engine but have different SLAs and different cost profiles.

03Placement & ownership

The enrichment layer sits immediately after normalization and before search/graph indexing. It is the fifth stage of the generic data pipeline:

  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
  │  Source  │──▶│ Normalize│──▶│  Persist │──▶│  ENRICH  │──▶│ Index +  │
  │ adapters │   │          │   │  (NEW)   │   │ (→ACTIVE)│   │ fan-out  │
  └──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘

Records enter persistence in their initial lifecycle state (NEW). The enrichment service consumes normalization.complete events, runs the enrichment, writes the result back, and publishes enrichment.complete — which triggers the lifecycle transition to ACTIVE and the downstream indexing fan-out.

Service layout

apps/enrichment-service/
  src/

    routes.ts           # 5 HTTP endpoints

    service.ts          # enrichSingle / enrichBatch / processBatchResults
    budget.ts           # checkTokenBudget — called before every LLM call
    providers/
      provider-a.ts     # External lookup adapters
      provider-b.ts
      circuit-breaker.ts
    prompts/
      system-prompts.ts # Per-type system prompts (cached)
      corpus.ts         # Large reference corpus (cached)
      builders.ts       # buildPrompt(record, providerResults)
    schema.ts           # EnrichmentOutputSchema (Zod / Pydantic / JSON Schema)
    repository.ts       # Persistence
    queue.ts            # Real-time worker, batch worker, batch poller
    README.md

Unlike normalization (which is a shared package consumed in-process by every other service), enrichment is a real service. The reason: it has external dependencies (LLM provider, third-party APIs), it owns expensive state (token budget, prompt cache), and its failure modes (rate limits, provider outages) need to be isolated from the rest of the system.

04The three enrichment patterns

All three patterns share the same engine. They differ in how the work is queued and what the SLA looks like.

PATTERN 1
Synchronous

Real-time + cache

For human-triggered single lookups. A user clicks a record in the UI and expects sub-second feedback. The hot path is engineered to make the cache hit in <50ms, and the cold path (cache miss) to complete in 1-3 seconds.

async function enrichSingle(record: CanonicalRecord, tenantId: string): Promise<EnrichmentOutput> {
  // 1. Cache check — sub-millisecond hit on Redis
  const cacheKey = `enrichment:${record.type}:${record.normalized_value}`;
  const cached = await redis.get(cacheKey);
  if (cached) return EnrichmentOutputSchema.parse(JSON.parse(cached));

  // 2. Budget gate — never call the LLM without checking
  const budget = await checkTokenBudget(tenantId, 'enrichment');
  if (!budget.allowed) {
    if (budget.fallbackToCheaperModel) return enrichWithCheapModel(record, tenantId);
    throw new BudgetExceededError();
  }

  // 3. Parallel external provider calls — zero LLM cost
  const providerResults = await Promise.allSettled([
    providerA.lookup(record),
    providerB.lookup(record),
    providerC.lookup(record),
  ]);

  // 4. LLM call with prompt caching
  const llmResponse = await llm.complete({
    model: MODELS.default,
    system: [
      { text: SYSTEM_PROMPTS[record.type], cache: 'ephemeral' },
      { text: REFERENCE_CORPUS,             cache: 'ephemeral' },
    ],
    user: buildPrompt(record, providerResults),
    maxOutputTokens: 1024,
  });

  // 5. Schema validation — never store unvalidated output
  const validated = EnrichmentOutputSchema.parse(extractJson(llmResponse));

  // 6. Cache with type-specific TTL
  await redis.setex(cacheKey, CACHE_TTL[record.type], JSON.stringify(validated));

  // 7. Token usage tracking
  await trackTokenUsage({ tenantId, module: 'enrichment', usage: llmResponse.usage });

  return validated;
}
PATTERN 2
Asynchronous bulk

Batch API enrichment

For non-real-time enrichment of N or more records. Most LLM providers offer a batch API that runs work asynchronously over hours in exchange for a 50% cost reduction. This is the right path for any feed-driven ingest where the SLA is "complete within an hour" rather than "complete within a second."

The flow:

  1. Ingestion service buffers records until it has at least N (configurable, typically 10-50)
  2. Enrichment service builds a single batch request with one entry per record
  3. Batch is submitted, ID is stored, and a polling worker is scheduled
  4. Worker polls every minute for up to one hour
  5. On completion, results are validated one-by-one and written back to the corresponding records
async function enrichBatch(records: CanonicalRecord[], tenantId: string): Promise<string> {
  if (records.length < BATCH_MIN_SIZE) throw new BatchTooSmallError();

  // Budget for the entire batch upfront
  const budget = await checkTokenBudget(tenantId, 'enrichment', records.length * AVG_TOKENS);
  if (!budget.allowed) throw new BudgetExceededError();

  // Build batch with one request per record, sharing cached system prompt
  const requests = records.map((r, idx) => ({
    custom_id: `enrich-${r.id}-${idx}`,
    params: {
      model: MODELS.default,
      system: [{ text: SYSTEM_PROMPTS[r.type], cache: 'ephemeral' }],
      user: buildPrompt(r),
      maxOutputTokens: 1024,
    },
  }));

  const batch = await llm.batches.create({ requests });
  await redis.setex(`batch:${batch.id}`, 86400, JSON.stringify({
    tenantId, recordIds: records.map(r => r.id), submittedAt: new Date().toISOString(),
  }));

  // Schedule polling worker — 60s interval, up to 60 attempts (1 hour SLA)
  await batchQueue.add('batch-poll', { batchId: batch.id, tenantId }, {
    delay: 60_000, attempts: 60,
    backoff: { type: 'fixed', delay: 60_000 },
  });

  return batch.id;
}
Why batch is the default, not the exception

For feed-driven ingest, the 50% cost reduction is the difference between affordable and unaffordable at scale. The hourly latency rarely matters because nobody is staring at the screen waiting. Real-time should be reserved for cases where a human is actually waiting for the answer.

PATTERN 3
Multi-step

Agentic enrichment

For complex investigations that require more than one LLM call to complete. The model is given a set of tools — graph traversal, search, related-record lookup, fact retrieval — and runs an iterative loop: hypothesize, query, evaluate, refine, conclude.

This pattern is reserved for high-value workflows where a single-shot enrichment isn't enough. Examples: tracing how a record connects to others several hops away, building a timeline of related events, or producing a narrative summary that requires looking up multiple supporting records.

Cost discipline is even more important here than in single-shot enrichment, because agentic loops can spiral. Two guardrails:

  • Hard step limit. The loop exits after N tool calls regardless of completion. N is small (5-10) and configurable per workflow.
  • Per-investigation token budget. A separate budget bucket is allocated per investigation. When it runs out, the loop stops and returns whatever it has.

Agentic enrichment should always be human-initiated and optionally human-reviewed before any of its outputs propagate downstream.

05External provider orchestration

External providers are called before the LLM, in parallel, and their output is fed into the LLM prompt as structured context. This is the single biggest cost reduction lever in the entire system: the LLM doesn't waste tokens looking up facts it could be told.

Parallel fan-out

const providerResults = await Promise.allSettled([
  providerA.lookup(record),
  providerB.lookup(record),
  providerC.lookup(record),
  providerD.lookup(record),
]);

const consolidated = {
  providerA: providerResults[0].status === 'fulfilled' ? providerResults[0].value : null,
  providerB: providerResults[1].status === 'fulfilled' ? providerResults[1].value : null,
  providerC: providerResults[2].status === 'fulfilled' ? providerResults[2].value : null,
  providerD: providerResults[3].status === 'fulfilled' ? providerResults[3].value : null,
  unavailable: providerResults
    .map((r, i) => r.status === 'rejected' ? PROVIDER_NAMES[i] : null)
    .filter(Boolean),
};

The unavailable array is included in the LLM prompt so the model knows what data was missing. This is what makes the system degrade gracefully: a flapping provider doesn't take down enrichment, the LLM just gets less context and is told so.

Circuit breakers per provider

Wrap every provider call in a circuit breaker (Hystrix, Opossum, Resilience4j — pick your stack). Set per-provider timeouts at 2 seconds maximum, and trip the breaker after 5 consecutive failures. A tripped breaker returns null immediately for the next 60 seconds without making the network call. This is what prevents one slow provider from serializing the whole pipeline.

Per-provider rate limits

Many free-tier external APIs have aggressive rate limits (e.g., 4 requests per minute). The circuit breaker doesn't catch this — the call succeeds, just slowly. Add a per-provider token bucket at the adapter level so requests queue locally rather than getting rejected by the upstream.

06Prompt caching strategy

Modern LLM APIs support prompt caching — marking parts of the prompt as cacheable so subsequent calls within a window pay only for the changing portion. Used correctly, this is a 60-90% reduction in input token cost.

What to cache

Two blocks per request:

  1. The system prompt — typically 1-2K tokens, one per record type. Identical across all calls of the same type.
  2. The reference corpus — large body of background knowledge the LLM needs (taxonomies, classification frameworks, examples). Often 100K+ tokens. Identical across all calls regardless of type.
const response = await llm.complete({
  model: MODELS.default,
  system: [
    { text: SYSTEM_PROMPTS[record.type], cache: 'ephemeral' },  // ~1-2K tokens
    { text: REFERENCE_CORPUS,             cache: 'ephemeral' }, // ~200K tokens
  ],
  user: buildPrompt(record, providerResults),                    // ~500 tokens — the only changing part
  maxOutputTokens: 1024,
});

After the first call in the cache window, subsequent calls pay only for the user message — the cached blocks are billed at 10% of normal input cost (provider-dependent).

Target hit rate

Target: 80%+ cache hit rate on the cached blocks. Below this number, something is wrong — either the cache window is too short for your traffic pattern, or the cached blocks are changing more often than they should.

Instrument it

Most LLM APIs return separate counters for cached vs uncached input tokens (cache_read_input_tokens, cache_creation_input_tokens). Track them separately in your token usage table, dashboard the ratio, and alert if it drops below your target. This is the single most important metric for enrichment cost.

Don't put dynamic content in cached blocks

If you templatize the system prompt with the tenant name or current date, the cache will never hit. Cached blocks must be byte-identical across calls. Put dynamic content in the user message.

07Token budget enforcement

Every tenant has a monthly token budget per module. Every LLM call passes through a budget check before it issues the request. No exceptions, no escape hatches.

Three-tier enforcement

Budget usedActionUser experience
< 80% Normal — use default model No visible change
80–90% Warning event published, dashboard banner shown Admin sees warning, no degradation
90–100% Automatic fallback to cheaper model Slightly lower quality, no failures
≥ 100% Hard stop — return budget-exceeded error Admin notified, user sees clear error 
async function checkTokenBudget(tenantId, module, estimatedTokens = 2000) {
  const key = `budget:${tenantId}:${module}:${currentMonth()}`;
  const used = Number(await redis.get(key) ?? 0);
  const limit = await getTenantLimit(tenantId, module);
  const usedPct = (used / limit) * 100;

  if (usedPct >= 80 && usedPct < 90)
    await publishEvent('budget.warning', { tenantId, module, usedPct });

  if (usedPct >= 90 && usedPct < 100)
    return { allowed: true, fallbackToCheaperModel: true };

  if (limit - used < estimatedTokens) {
    await publishEvent('budget.exceeded', { tenantId, module, usedPct });
    return { allowed: false, fallbackToCheaperModel: false };
  }
  return { allowed: true, fallbackToCheaperModel: false };
}

Per-module budgets

Different modules have different budgets because their workloads have different value-per-token. Typical allocation:

ModuleDefault monthly budgetRationale
Enrichment500K tokensHighest volume — every record passes through
Correlation200K tokensLower volume, runs on clusters
Investigation / hunting100K tokensHuman-initiated only, low volume

Budgets are per-tenant overrideable from an admin panel. Enterprise customers get higher caps.

08Model tiering

Not every task needs the most expensive model. Tier the models against the tasks:

TierUse forWhy
Reasoner
(mid-tier)		 Single-record enrichment, multi-step reasoning, cluster analysis Best balance of quality and cost. The default for any task that requires actual reasoning.
Classifier
(small/fast)		 Triage, classification, tag extraction, severity labeling, summarization, budget-fallback target 5-10x cheaper than the reasoner. Quality is 80-90% as good for narrow extraction tasks.
Heavyweight
(top-tier)		 Development conversations only — never in production runtime Too expensive for per-record use. Reserved for offline tasks where a human is reading the output.

The triage pattern

One of the highest-ROI uses of model tiering: a two-stage cascade where the cheap classifier filters out work the reasoner doesn't need to do.

  1. Stage 1 (cheap classifier): "Is this record worth deeply enriching?" — yes/no plus a confidence score.
  2. Stage 2 (expensive reasoner): Only runs on records the classifier flagged as worth processing.

If 80% of incoming records are not worth deep enrichment, this single pattern cuts the expensive model bill by ~80% with negligible quality loss.

09Enforced output schema & reasoning trail

The LLM is instructed to return JSON matching a strict schema. The response is validated at runtime. If validation fails, the result is rejected and either retried or quarantined — never coerced, never stored as free text.

The schema

interface EnrichmentOutput {
  // Core scoring
  score:           number;          // 0-100 risk/relevance/priority score
  confidence:      number;          // 0.0-1.0 model self-confidence

  // Classification
  categories:      string[];        // Discrete labels from a controlled vocabulary
  attributions:    string[];        // Named entities the record is linked to

  // Reasoning trail — the auditability differentiator
  reasoning: {
    score_justification: string;    // 2-3 sentences citing specific evidence
    evidence_sources:    string[];  // Concrete evidence items used
    uncertainty_factors: string[];  // What would change this score
  };

  // Action guidance
  recommended_actions: string[];

  // Negative-result handling
  is_false_positive:    boolean;
  false_positive_reason: string | null;

  // Provenance
  schema_version:  '1.0';
}

The reasoning trail is the differentiator

Most enrichment systems output a number with no explanation. This is unusable in any context where a human will be asked to defend the result — a security review, a compliance audit, an analyst walkthrough. The reasoning trail solves this by making every score self-documenting:

  • score_justification — 2-3 sentences explaining the score in concrete terms, citing specific evidence the LLM used.
  • evidence_sources — list of the concrete items (provider results, classification taxonomies, related records) that informed the score.
  • uncertainty_factors — what the model is unsure about. This is the most undervalued field in the schema. A model that admits uncertainty is more trustworthy than one that doesn't.

Validation enforcement

const result = EnrichmentOutputSchema.safeParse(extractJson(llmResponse));
if (!result.success) {
  // Optional: one retry with the validation errors fed back to the model
  // Otherwise: quarantine and alert
  await quarantine.store({ rawResponse: llmResponse, errors: result.error.issues });
  throw new SchemaValidationError(result.error);
}

Add a Zod refinement that requires reasoning.evidence_sources to be non-empty. If the LLM returns an empty array, that's a hallucinated score — reject it.

Why a strict schema beats free-text output

A strict schema turns the LLM into a typed function call. Downstream code can read the fields with confidence, the UI can render them in dedicated slots, and bugs surface as validation errors instead of garbled displays. Free-text output looks more flexible but creates a permanent integration tax.

10Composite final scoring

The LLM's score is one signal. The final score stored on the record combines the LLM output with the external provider signals using documented weights:

finalScore = (providerA_score × 0.35)
           + (providerB_score × 0.30)
           + (llm_score       × 0.35)

Weights sum to 1.0. When a provider is unavailable, weights are renormalized over the remaining signals (LLM alone = llm_score × 1.0).

Why combine signals at all? Because any single signal — including the LLM — has blind spots. The LLM might miss a pattern that an external reputation database catches. The reputation database might be outdated on a record the LLM has fresher context on. Composite scoring is what makes the final number more robust than any individual input.

This composite final score then feeds back into the normalization layer's composite confidence formula (see the Normalization Architecture Blueprint, section 06) — the two layers reinforce each other.

11Integration contract

The enrichment service consumes one event and produces one event. Same narrow contract pattern as normalization.

DirectionChannelPayload
consumes normalization.complete { recordId, type, tenantId }
produces enrichment.complete { recordId, type, tenantId, score, confidence }
produces enrichment.failed { recordId, errorCode, errorDetails }
produces budget.warning { tenantId, module, usedPct }
produces budget.exceeded { tenantId, module, usedPct }

The enrichment.complete event is what triggers the lifecycle transition NEW → ACTIVE on the record (see the Normalization Blueprint, section 08). A record stuck in NEW state for more than 24 hours is a tripwire — something is wrong with the enrichment pipeline and an alert should fire.

12Reuse checklist for a new project

To drop this pattern into a new domain, work through these questions in order. Each one corresponds to a section above.

  1. What does "enrichment" mean for your domain? Write down the specific decisions the layer should produce — score, classification, action recommendation, narrative summary. This determines your output schema.
  2. Which external providers add context for free or cheap? List them. For each, write the adapter, the timeout, and the rate limit. This becomes section 05.
  3. What goes in the system prompt? One per record type. Keep it under 2K tokens. Test it with 20 hand-picked examples before committing. This becomes section 06.
  4. Is there a large reference corpus? Taxonomies, classification frameworks, controlled vocabularies. If yes, this is your second cached block. If no, skip it — but instrument cache hit rate anyway.
  5. What is your monthly token budget per tenant per module? Pick numbers, even if they're guesses. You can tune later. This becomes section 07.
  6. What models are available, and how do they tier? Identify the reasoner, the classifier, and the heavyweight. Document which is used for which task. This becomes section 08.
  7. What is the output schema? Draft it in your runtime validator of choice. Make the reasoning trail mandatory. This becomes section 09.
  8. What are the composite scoring weights? Pick a starting point that sums to 1.0. Plan to refit against ground truth after 30 days. This becomes section 10.
  9. What is the SLA for real-time vs batch? Real-time should be sub-3-second. Batch should be sub-1-hour. Anything else is wrong.
  10. What is the minimum batch size? Lower bound is 10. Higher is better for cache reuse, as long as it doesn't blow your SLA.
  11. What is the cache TTL per record type? Volatile types: minutes to hours. Stable types: days to weeks. Document the rationale per type.
  12. What is the quarantine path for schema validation failures? Build a simple admin view for inspecting and reprocessing failed enrichments.

Twelve answers, then build. The layer takes about a week to get running and another week to tune cache hit rate and budget thresholds against real traffic.

13Anti-patterns to avoid

Calling the LLM without a cache check

Every miss that should have been a hit is wasted money and wasted latency. Every single LLM call must be preceded by a Redis check. No exceptions, even for "small" calls — they're never small at scale.

Calling the LLM without a budget check

A runaway loop or a misconfigured ingest can burn a month's budget in an hour. The budget check is the only thing standing between you and a surprise invoice. It must be unconditional.

Storing free-text LLM output

"We'll parse it later" never happens. Free-text output means downstream code is forever doing string matching against model responses, breaking every time the model changes. Strict schema, validated at the gate.

Score with no reasoning trail

A number that nobody can explain is a number nobody can defend. The first time a human asks "why did the system score this an 87?" and you can't answer, you've lost their trust permanently. Reasoning trail is mandatory.

Putting dynamic content in cached blocks

Templating the system prompt with the tenant name, current date, or record ID destroys the cache. The cached block must be byte-identical across calls. Dynamic content goes in the user message.

Synchronous fan-out over external providers

Calling provider A, then provider B, then provider C in sequence multiplies their latency. Always parallel via Promise.allSettled (or your stack's equivalent), always with per-provider circuit breakers.

Real-time as the default, batch as the exception

For feed-driven ingest, this gets the cost-vs-latency tradeoff exactly backwards. Real-time should be reserved for human-triggered actions. Everything else should batch.

Using the heavyweight model in production runtime

The top-tier model is for development conversations and offline tasks where a human is reading the output. Putting it in the per-record hot path will bankrupt the project. The mid-tier reasoner is the production default.

No instrumentation on cache hit rate

If you don't track cache_read_input_tokens / total_input_tokens, you can't tell whether your prompt caching is actually working. The first time you check the dashboard you'll discover it's been at 12% for three months. Instrument from day one.
