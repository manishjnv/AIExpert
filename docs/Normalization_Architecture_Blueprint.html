Data Normalization Engine

01Why a normalization layer exists

Any platform that ingests data from more than one source eventually faces the same problem: every source uses different field names, different value formats, different timestamps, different confidence scales, and different ideas of what counts as a duplicate. Without a single chokepoint that converts everything to one canonical shape, downstream code rots into an ever-growing pile of if source === 'X' branches.

The normalization layer is that chokepoint. Its job is narrow but absolute: nothing reaches storage until it has been transformed into the canonical form. Every search index, every graph node, every alert rule, every API response can then assume one schema. The cost of building it once is paid back the first time you add a third source.

The one rule that makes everything else work

Normalization is a shared package (a library every module imports), not a microservice with an HTTP boundary. Microservices add latency and a network failure mode for what should be a pure function call. The thin "normalization service" that exists is just a queue worker — the actual normalization logic lives in the package and is called directly by every other service.

02Core principles

Pure functions

Every method in the normalizer is stateless. No database calls, no network I/O, no logging side effects. This makes it trivially testable and safe to call from any context (worker, hot path, batch job).

Fail loudly, never silently

Malformed input throws a typed error. Never coerce garbage into a "best guess" record — that data corrupts every downstream system that trusts the canonical schema.

Single source of truth for schemas

Canonical schemas live in one shared types package and are imported everywhere. The package is the API contract. Bumping a schema version requires a migration, not an ad-hoc field add.

Determinism over cleverness

Given the same input, the normalizer must always produce byte-identical output — including the dedup hash. This is what allows safe upserts, replay, and audit.

Priority order is part of the spec

When type detection or rule matching has overlapping patterns, the order in which they are evaluated is part of the documented behavior, not an implementation detail. Test it.

Versioned schemas, never silent drift

Every canonical record carries a schemaVersion field. When the shape changes, you bump the version and ship a migration — never mutate fields in place.

03Placement & ownership

The normalization layer sits between ingestion and persistence. It is the third stage of a generic data pipeline:

  ┌──────────┐   ┌───────────┐   ┌───────────────┐   ┌──────────┐   ┌──────────┐
  │  Source  │──▶│  Ingest   │──▶│  NORMALIZE    │──▶│  Persist │──▶│ Index +  │
  │ adapters │   │   queue   │   │  (this layer) │   │  (upsert)│   │ fan-out  │
  └──────────┘   └───────────┘   └───────────────┘   └──────────┘   └──────────┘

It owns the boundary between "raw, source-shaped data" and "canonical, downstream-ready data." Everything to the left of it is allowed to be messy. Everything to the right of it is allowed to assume one shape.

Package layout

packages/shared-normalization/
  src/
    normalizer.ts        # The main class — N stateless methods
    confidence.ts        # Composite scoring formula
    lifecycle.ts         # State machine + valid transitions
    dedupe.ts            # Hash-key construction
    detectors/
      type-detector.ts   # Priority-ordered classifier
    cleaners/
      value-cleaner.ts   # Trim, lowercase, canonicalize
    index.ts             # Barrel export
  __tests__/
  README.md

The thin service that wraps it (a queue worker) lives elsewhere — apps/normalization-service/ — and is roughly 100 lines: pull job, call normalizer.normalize(raw), push result to next queue. The interesting code is all in the package.

04The 10 phases of normalization

Every record passes through these phases in order. Each phase has one responsibility, one input shape, and one output shape. Skipping a phase is a bug.

PHASE 1
Detect

Type detection & classification

Given a raw value or record, decide what kind of entity it represents. This is almost always implemented as a priority-ordered list of pattern matchers — most specific first, most general last. The order is a contract: testing it requires explicit cases for every adjacent pair of types that could be confused.

const ORDERED_PATTERNS: Array<[EntityType, RegExp]> = [
  ['most_specific',     /^...$/],
  ['next_specific',     /^...$/],
  // ...
  ['most_general',      /^...$/],
];

function detectType(value: string): EntityType | null {
  for (const [type, pattern] of ORDERED_PATTERNS) {
    if (pattern.test(value)) return type;
  }
  return null;  // Unknown — caller decides whether to reject or quarantine
}

Watch out for: patterns where one is a substring length of another (e.g., a 40-char hex string is both a valid SHA-1 and a valid prefix of a SHA-256). Order matters more than cleverness.

PHASE 2
Clean

Value cleaning & canonicalization

Strip the cosmetic differences between sources. Lowercase, trim whitespace, remove protocol prefixes when they're not meaningful, collapse trailing slashes, normalize unicode, decode percent-encoding. The output is the form that will be stored in normalized_value and used for the dedup hash — so it must be deterministic.

Keep the original raw form too, in a value field. Display layers show value; dedup and search use normalized_value. Two fields, one purpose each.

PHASE 3
Map

Schema mapping (raw → canonical)

Translate source-shaped fields into canonical fields. This is the part where each source needs its own adapter — but the adapters are tiny, because they only do field renaming and type coercion. They never make decisions about validity or merging.

// One adapter per source
function fromSourceA(raw: SourceARecord): CanonicalRecord {
  return {
    type:        raw.kind,
    value:       raw.observable,
    firstSeen:   normalizeTimestamp(raw.first_observed_ts),
    confidence:  normalizeConfidence(raw.score),
    sourceRef:   { sourceId: 'source-a', recordId: raw.id },
    rawData:     raw,           // Keep original for audit
  };
}

The full canonical record is then passed to subsequent phases — the adapter is done.

PHASE 4
Coerce

Scalar normalization

Three scalar types reliably differ between sources and need their own normalizers:

  • Timestamps — accept ISO 8601, Unix seconds, Unix milliseconds, RFC 2822, source-specific strings. Always output ISO 8601 UTC.
  • Confidence / score — accept 0-1 floats, 0-100 ints, "low"|"medium"|"high" strings, source-specific scales. Always output 0-100 integer.
  • Enums — accept any case, with or without prefix, with synonyms. Always output the canonical enum value, never null. Default to the most conservative enum if input is missing.

These are the three places silent data corruption hides. Test them ruthlessly.

PHASE 5
Validate

Schema validation

Run the canonical record through a runtime validator (Zod, Pydantic, JSON Schema, whatever your stack uses). This is the gate: if the record fails validation, it never reaches persistence. Failed records go to a quarantine table with the validation error attached so a human can review them later.

const result = CanonicalRecord.safeParse(record);
if (!result.success) {
  await quarantine.store({ raw, errors: result.error.issues });
  throw new ValidationError('NORMALIZATION_FAILED', result.error);
}

The validator is the same schema definition that the API layer uses for requests and responses — one source of truth, no drift.

PHASE 6
Filter

Privacy & safety filtering

Drop records that should never be stored. Common examples: private IP ranges, internal hostnames, PII that violates retention policy, records flagged by upstream as "do not propagate." This phase exists because filtering at the source is unreliable — you'll always get one source that forgot.

Filtered records are logged (with reason) but not stored. They are not the same as quarantined records, which are broken rather than unwanted.

PHASE 7
Hash

Dedup key construction

Build a deterministic SHA-256 hash from the fields that define identity. For most platforms this is:

dedupeKey = sha256(`${type}:${normalized_value}:${tenantId}`)

The hash is stored as a unique index on the persistence layer, which makes the next phase's upsert atomic. Identity fields must be lowercase and trimmed before hashing — any drift here means duplicates will leak through.

Tripwire: log any case where two different raw values produce the same hash. By construction it should be impossible, but logging it catches normalization bugs the moment they appear.

PHASE 8
Score

Initial confidence computation

Compute a baseline confidence score from the signals available at normalization time — typically source reliability and corroboration count. Downstream phases (enrichment, AI analysis, community feedback) will update this score later, but the record needs some confidence value to enter the system.

See section 06 for the full composite formula. The principle: confidence is never a single number from a single source.

PHASE 9
Persist

Atomic upsert

The persistence call is always an upsert keyed on dedupeKey. On insert, the record is created in its initial lifecycle state. On conflict, the existing record is updated:

  • lastSeen moves forward
  • sightings counter increments
  • sourceRefs array appends the new source
  • confidence recomputes (corroboration count went up)
  • Lifecycle state may transition (e.g., aging record re-activates)
await db.record.upsert({
  where:  { dedupeKey },
  create: { ...canonical, sightings: 1, state: 'NEW' },
  update: {
    lastSeen:    canonical.lastSeen,
    sightings:   { increment: 1 },
    sourceRefs:  { push: canonical.sourceRefs[0] },
    confidence:  recomputeComposite(...),
    state:       maybeTransition(currentState, 'sighted'),
  }
});
PHASE 10
Emit

Event publication

Publish a single canonical event to the bus: normalization.complete. The payload is the persisted record's ID, type, and tenant — not the full record. Downstream consumers (search indexer, graph builder, enrichment service, alert engine) listen for this event and pull what they need.

This is the only way downstream systems learn about new records. Direct enqueueing into downstream queues from any other module is forbidden — it's the second-most-common cause of pipeline drift after per-module normalizers.

05Canonical schema pattern

Every canonical record shares a common envelope, regardless of entity type. The envelope makes the entire downstream stack — storage, search, graph, audit — possible to build generically.

interface CanonicalRecord<T = unknown> {
  // Identity
  id:            string;          // UUID, generated at normalization time
  tenantId:      string;
  type:          EntityType;      // Discriminator for the typed payload
  value:         string;          // Original human-readable form
  normalized_value: string;       // Canonical form (lowercased, trimmed, etc.)

  // Provenance
  sourceRefs:    Array<{
    sourceId:    string;
    recordId:    string;
    ingestedAt:  string;          // ISO 8601 UTC
  }>;
  rawData:       T;                // Original payload, for replay/audit

  // Temporal
  firstSeen:     string;          // ISO 8601 UTC
  lastSeen:      string;
  normalizedAt:  string;

  // Quality
  confidence:    number;          // 0-100 composite
  severity:      Severity;        // INFO | LOW | MEDIUM | HIGH | CRITICAL
  tlp:           TLP;             // Or your domain's classification enum

  // Classification
  tags:          string[];
  labels:        string[];

  // Lifecycle
  state:         LifecycleState;  // See section 08
  sightings:     number;

  // Versioning
  schemaVersion: '1.0';           // Hardcoded — changing requires migration
}

Specific entity types extend this envelope with their own typed fields. The envelope itself never changes between types — only the typed payload differs.

Why rawData is non-negotiable

Storing the original source payload looks wasteful until the day you discover a normalization bug and need to re-process six months of records. With rawData you can replay normalization in minutes. Without it, the bug is permanent.

06Composite confidence formula

Confidence should never be a single number from a single source. The formula combines multiple independent signals with weights, then applies time decay so stale records automatically de-prioritize.

interface ConfidenceInputs {
  sourceReliability:    number;   // 0-100, source's historical accuracy
  corroborationCount:   number;   // # independent sources reporting the same record
  enrichmentConfidence: number;   // 0-100, from downstream enrichment pass
  communityScore:       number;   // 0-100, optional analyst/community rating
  ageDays:              number;   // days since firstSeen
}

const WEIGHTS = {
  sourceReliability:    0.25,
  corroboration:        0.25,
  enrichmentConfidence: 0.30,
  communityScore:       0.20,
} as const;

function computeCompositeConfidence(inputs: ConfidenceInputs): number {
  // Corroboration capped to prevent feed-stuffing manipulation
  const corroborationBonus = Math.min(100, inputs.corroborationCount * 20);

  const base =
    inputs.sourceReliability    * WEIGHTS.sourceReliability +
    corroborationBonus          * WEIGHTS.corroboration +
    inputs.enrichmentConfidence * WEIGHTS.enrichmentConfidence +
    inputs.communityScore       * WEIGHTS.communityScore;

  // Exponential decay with ~125 day half-life
  const decayFactor = Math.exp(-inputs.ageDays / 180);

  return Math.round(Math.max(0, Math.min(100, base * decayFactor)));
}

Why these weights: the highest single weight (0.30) goes to enrichment because it has the most context, but no signal exceeds a third of the total. Corroboration is capped because otherwise a single dishonest source operating five accounts would dominate the score.

Why exponential decay: a record that hasn't been re-sighted in six months is roughly 37% of its original confidence — automatically de-prioritized in alerts and dashboards without anyone needing to manually expire it. The 180-day denominator gives a half-life around 125 days; tune for your domain.

Tune the weights with real data

The starting weights are a sensible default. Once you have 30+ days of records with confirmed true/false outcomes, refit the weights with logistic regression against ground truth. This is the single biggest accuracy lever and the one most teams never touch.

07Deduplication strategy

Two layers of dedup, working together:

Layer 1 — Exact match (in normalization)

The composite SHA-256 hash. Catches the case where the same record arrives from multiple sources with byte-identical canonical form. Sub-millisecond, atomic via unique index.

Coverage: ~80% of duplicates.

Layer 2 — Semantic match (in ingestion, before normalization)

For records that mean the same thing but differ slightly in form — typos, alternate spellings, near-duplicate text. Three-stage cascade:

  1. Bloom filter — sub-ms exact-match pre-check on normalized value
  2. Vector similarity — pgvector / Faiss cosine similarity on embeddings, threshold ~0.92
  3. LLM arbitration — only for ambiguous cases in the 0.85-0.95 band

Layer 1 lives inside the normalization package. Layer 2 lives upstream in the ingestion service because it needs to happen before normalization runs (no point normalizing a record you're about to drop). Both write to the same sightings counter when they detect a hit.

08Lifecycle state machine

Every canonical record follows a strict state machine. Transitions are enforced in code — no state can be skipped, and no raw UPDATE state = ... statement is allowed anywhere outside the lifecycle module.

  NEW ──▶ ACTIVE ──▶ AGING ──▶ EXPIRED ──▶ ARCHIVED
   │        │                              │
   │        └──▶ FALSE_POSITIVE ────────────┘
   │                                        │
   └──▶ REVOKED ────────────────────────────┘
TransitionTriggerType
NEW → ACTIVEEnrichment pass completes successfullyAutomatic
ACTIVE → AGINGNo new sighting for N days (configurable per type)Cron
AGING → EXPIREDNo new sighting for 2N daysCron
EXPIRED → ARCHIVED3N days after expiry — moved to cold storageCron
AGING/EXPIRED → ACTIVENew sighting arrives — re-activationAutomatic
* → FALSE_POSITIVEUser marks via UIManual
* → REVOKEDSource retracts the recordManual / source-driven

Enforcement

const TRANSITIONS: Record<State, readonly State[]> = {
  NEW:            ['ACTIVE', 'REVOKED'],
  ACTIVE:         ['AGING', 'FALSE_POSITIVE', 'REVOKED'],
  AGING:          ['EXPIRED', 'ACTIVE'],
  EXPIRED:        ['ARCHIVED', 'ACTIVE'],
  ARCHIVED:       [],
  FALSE_POSITIVE: ['ARCHIVED'],
  REVOKED:        ['ARCHIVED'],
} as const;

function transitionState(current: State, target: State): State {
  if (!TRANSITIONS[current].includes(target)) {
    throw new InvalidTransitionError(current, target);
  }
  return target;
}

Add a custom lint rule that bans direct state updates anywhere outside the lifecycle module. This single rule eliminates an entire class of "record stuck in wrong state" bugs.

09Integration contract

The normalization layer publishes exactly one event downstream and consumes exactly one event upstream. This narrow contract is what keeps the rest of the system loosely coupled.

DirectionChannelPayload
consumes ingestion.normalize { rawRecord, sourceMeta, tenantId }
produces normalization.complete { recordId, type, tenantId, dedupeKey }
produces normalization.failed { rawRecord, errorCode, errorDetails, tenantId }

Every downstream system — search indexer, graph builder, enrichment, alerting, audit log — subscribes to normalization.complete. None of them ever calls the persistence layer directly to find new records, because polling is brittle and event-driven is replayable.

Replay-ability for free

Because downstream systems learn about records exclusively through events, you can rebuild any downstream system from scratch by replaying the event log. No special re-indexing endpoints, no migration scripts — just replay.

10Reuse checklist for a new project

To drop this pattern into a new domain, work through these questions in order. Each one corresponds to a phase or component above.

  1. What are your entity types? List them. For each, write down the most distinctive pattern that identifies it. Order by specificity, most specific first. This becomes Phase 1.
  2. What does "canonical" mean for each value? Lowercase? Trim? Strip a prefix? Decode encoding? Document one rule per type. This becomes Phase 2.
  3. What sources will you ingest from? For each, write a thin adapter that maps source-shaped fields to canonical fields. Adapters do nothing else. This becomes Phase 3.
  4. What scalar formats vary across sources? Timestamps and confidence almost always do. List the variants you'll see and write a single normalizer for each. This becomes Phase 4.
  5. Define the canonical schema in your runtime validator of choice. Make it the same definition the API uses. This becomes Phase 5.
  6. What should never be stored? Privacy filters, internal-only flags, retention exclusions. This becomes Phase 6.
  7. What fields define identity? Usually type + normalized_value + tenantId. Hash them. This becomes Phase 7.
  8. What signals contribute to confidence? Pick 3-5 independent signals and assign weights summing to 1.0. Choose a decay half-life appropriate to your domain. This becomes Phase 8.
  9. Write the upsert. Atomic, keyed on the dedup hash, increments sightings. This becomes Phase 9.
  10. Define your lifecycle states. Most domains need NEW → ACTIVE → AGING → EXPIRED → ARCHIVED plus FALSE_POSITIVE and REVOKED. Build the transition map. This becomes lifecycle (section 08).
  11. Pick the event channel name. One in, one out, plus a failure channel. This becomes the integration contract (section 09).
  12. Wire the quarantine table. Failed validations go here, not into the main store. Build a simple admin view to inspect and reprocess.

If you can answer all twelve questions in writing before any code is written, the implementation itself takes a few days. Most of the value is in the questions, not the code.

11Anti-patterns to avoid

Per-module normalizers

"Each service will normalize its own data" sounds modular but is the single biggest source of long-term pain. Field names drift, confidence scales drift, dedup hashes drift, and downstream systems can no longer trust any field. Always one shared package.

Normalizing inside the persistence layer

Putting normalization logic inside ORM hooks or database triggers makes it invisible, untestable, and impossible to replay. Normalization is application logic. It belongs in code that you can unit-test.

"We'll add validation later"

Validation added after the fact will discover that 30% of stored records don't pass, and now you have a migration problem. Validate at the gate from day one. Quarantine, never coerce.

Confidence as a single number from a single source

Storing the source's reported score as your confidence means you've outsourced your most important quality signal to the least trustworthy party. Always compose multiple signals with documented weights.

Free-form state changes

If any code anywhere can run UPDATE record SET state = ?, then sooner or later someone will set it to a value the rest of the system doesn't expect. Route every state change through the lifecycle function and ban the raw update with a lint rule.

Direct downstream enqueueing

"It's faster to push directly into the search index queue from my service" — yes, until two services do it differently and the search index has half the records the database has. One event channel out of normalization, no exceptions.

Schema mutation without versioning

Adding a field to the canonical schema without bumping the version means old and new records coexist with different shapes. Downstream code now needs if record.field branches forever. Bump the version, write the migration, do it once.
