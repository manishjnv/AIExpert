# Building AutomateEdge Solo

A free, self-paced platform that gives anyone a personalised, AI-curated
study plan to learn modern AI from scratch — then tracks their progress,
grades their practice repos, and hands them a verifiable certificate.

This is the honest version of the story. What I built, why I built it
that way, what I got wrong, and what I'd do differently if I started
again tomorrow.

## Why this exists

The AI learning content that already exists falls into two buckets.

Static roadmaps (roadmap.sh, Coursera landing pages) are one-size-fits-
all and drift within a quarter. The thing at the top of the "you should
learn this first" list is rarely the thing you should learn first in
April 2026. Nobody maintains them.

Paid cohort programs solve the currency problem but gate-keep on price
and schedule. If you work a full-time job and live in a timezone where
the live sessions fall at 2 AM, you are not the target.

AutomateEdge sits between them. The curriculum is generated by AI from
trending signals (university course catalogues, practitioner blogs,
arXiv categories), gets re-evaluated every quarter, and is free.
Learners enrol, get a 3–12 month plan tailored to their level and
goal, link their practice repos as they build, and graduate with a
signed credential they can show a recruiter.

## The stack

FastAPI + SQLAlchemy 2.0 async + SQLite, a vanilla-JS frontend with no
framework, and seven AI providers on a fallback chain with Gemini Flash
2.5 as the primary.

Every one of those choices is a boring choice for the same reason:
this is a solo project running on one VPS, and every layer I don't
add is one fewer thing I have to keep alive at 2 AM.

**SQLite over Postgres.** ~10 active users at launch, maybe 1,000
next year. SQLite handles that with its feet up. The entire database
is one file I can rsync off the server in a second. The day I need
row-level concurrency for more than a few hundred writes per second,
I'll switch. That day is not today.

**Vanilla JS over React.** The app is forms + fetch + a few DOM
updates. The original tracker was one static HTML file that worked
offline. Every feature since has been progressive enhancement on top
of that. I can read the entire frontend in one sitting. No bundler,
no transpile step, no `npm audit` surprises.

**Seven AI providers on a chain.** Gemini Flash 2.5 first (fast, cheap,
and I have paid credits), then Groq → Cerebras → Mistral → DeepSeek →
Sambanova → Anthropic. The chain matters because free tiers rate-limit
aggressively and one provider being down shouldn't take the app down.
A circuit breaker skips providers that have failed recently; a cost-
cap gate refuses calls that would exceed a per-day $ budget. OpenAI
is strictly embeddings-only (topic dedup) — the quality-per-dollar at
that tier doesn't justify adding it to the generation chain.

## The AI quality pipeline

The thing I'm most glad I built is the quality pipeline, because it
made the generated content not-embarrassing.

Every AI-generated template runs through: **generate → review → refine
→ validate → score**. The score is a weighted composite across 15
heuristic dimensions — Bloom's progression, theory/practice ratio,
project density, assessment quality, completeness, freshness,
prerequisites clarity, and so on. Most of the dimensions are regex +
structural checks (zero AI cost); a handful call out to the Gemini 2.5
Pro refiner for deep rewrites of weak weeks.

Templates that score ≥90 are *publishable*, not *published*. That
distinction used to be "auto-publish if ≥90", and that's how
AutomateEdge put a confidently-wrong 91-scoring curriculum in front of
users for about two days. From session 9, every auto-generated
template lands as a draft. An admin has to click Publish manually,
and that action stamps `last_reviewed_by` and `last_reviewed_on` into
`_meta.json`. Human in the loop, by policy. The scoring rubric is a
filter, not a judge.

## The certificate system

The certificate is the emotional payoff. A learner finishes all tasks,
links their repos, and gets a PDF with their name, the course title,
their stats, an HMAC-signed credential ID, a QR code pointing at the
public verify page, and an OpenGraph image they can share on LinkedIn.

The HMAC signing is the important part. The credential ID looks like
`AER-2604-abc12-def34`, and the final chunk is an HMAC-SHA256 over the
user ID, course key, and issue date, truncated to 8 hex chars. A
recruiter can paste the ID at `/verify/<id>` and the server re-derives
the HMAC from the database row — if it matches, the credential is
genuine. If someone hand-crafts an ID, the HMAC won't match. The
secret is separate from the JWT secret so we can rotate cert
signatures without invalidating sessions.

PDF rendering goes through WeasyPrint. The first shipped build crashed
on every render because WeasyPrint 62.3 has an incompatibility with
pydyf 0.11 that only triggers on certain layouts; pinning weasyprint
to 63.1 and pydyf to 0.11 fixed it. This is the kind of thing you
only discover by running the thing in prod.

## The leaderboard (gamification, done seriously)

The leaderboard started as a CSV-export nice-to-have and grew into
the single highest-engagement page on the site.

It ranks learners by XP:
`tasks × 10 + distinct_repos × 50 + streak_weeks × 20 + cert_bonus`,
where cert_bonus is 500 / 750 / 1000 for completion / distinction /
honors. Seven tiers (Apprentice 🥚 → AI Guru 🧙), each with a
colored chip and a mini XP-bar to the next tier. Achievement pills
for milestones (First Task, Triple Crown, 10-week Hot Streak).
Distinct repos, not repo-links — same repo linked to multiple weeks
counts once, so you can't farm XP by re-linking. Last-active
recency with a color code (green ≤ 48h, amber ≤ 1w, muted older).

All of this is derived from real database state. No fake numbers, no
placeholder users, no manual curation. If the leaderboard says you're
Top 5%, you are.

## What I'd do differently

**Auto-publish.** I'd never have shipped it. The "quality score is
above the threshold, so obviously it's fine" assumption is exactly
the class of mistake the threshold was supposed to prevent. Every
generation pipeline needs a human gate by default, even if the gate
is just "this person clicked a button".

**More integration tests earlier.** The unit tests stayed green
through three weeks of drift in the migrate-progress endpoint
(server-wins logic was refactored, the unit test was still asserting
the old response shape, the endpoint in prod had been quietly broken
for the migrate path since session 6). Session 9 finally added a
real end-to-end journey test. That should have been test #1, not
test #104.

**Settled on Gemini 2.5 Pro for refinement sooner.** I spent weeks on
Claude + batch API + prompt caching before accepting that for the
refinement workload specifically, Gemini Pro was the right tool.
Sticking with the "Claude is smarter, therefore Claude is better"
gut for weeks past the point I had evidence otherwise cost me both
time and money.

## What's next

- An admin UI for revoking certificates (currently DB-edit only).
- A recent-activity stream on the leaderboard, and rank-change
  arrows so learners can see their trajectory.
- An AI news feed and an AI jobs board — the "learn AI, then what"
  gap is the thing I get the most questions about from learners who
  are finishing their first course.
- More templates beyond the three generalist tracks. The quality
  pipeline is good enough now that the bottleneck is the publish-
  button click, not the generation.

## If you're thinking of doing something similar

Ship the boring version first. The first version of the tracker
worked offline, in one HTML file, and that constraint made every
later decision easier — progressive enhancement is a much nicer
place to start from than "let's refactor the frontend".

Build the observability before you need it. An admin dashboard with
cost caps + balance tracking + per-provider alerts is not a premature
optimisation when you're solo and your budget is $10 at each
provider. It's the difference between catching a runaway call pattern
at $1 and discovering it at $10.

And write the post-mortem at the end of every working session, even
if nobody reads it except the next session of yourself.
